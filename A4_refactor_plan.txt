7. EMBEDDED ARRAY IMPACT ON UPDATE OPERATIONS

OPERATION: Add Employee to Shift
--------------------------------
Before (Assignment 3):
  db.assignments.insertOne({
    employeeId: "E001",
    shiftId: "S001"
  })

After (Assignment 4):
  db.shifts.updateOne(
    { _id: ObjectId("...") },
    { $push: { employees: ObjectId("...") } }
  )

Considerations:
- Must ensure employee ObjectId exists before pushing
- Must check for duplicates (MongoDB $addToSet instead of $push?)
- Atomic operation - no separate collection to synchronize

OPERATION: Remove Employee from Shift
-------------------------------------
Before:
  db.assignments.deleteOne({
    employeeId: "E001",
    shiftId: "S001"
  })

After:
  db.shifts.updateOne(
    { _id: ObjectId("...") },
    { $pull: { employees: ObjectId("...") } }
  )

Considerations:
- $pull removes all instances (safe since we prevent duplicates)
- Atomic - no orphan records possible

OPERATION: Delete Employee (Cascade Effects)
--------------------------------------------
Before:
  1. Delete employee from employees collection
  2. Delete all assignments with that employeeId

After:
  1. Delete employee from employees collection
  2. Update ALL shifts where employees array contains this ObjectId:
     db.shifts.updateMany(
       { employees: employeeObjectId },
       { $pull: { employees: employeeObjectId } }
     )

Risk: If step 2 fails, shifts will have dangling ObjectId references.
Mitigation: Perform pull operation before delete, or use transaction.

OPERATION: Get All Employees for a Shift
------------------------------------------
Before:
  1. Find assignments with shiftId
  2. Extract employeeIds
  3. Find employees with those employeeIds

After:
  Option A (Aggregation):
    db.shifts.aggregate([
      { $match: { _id: shiftObjectId } },
      { $lookup: {
          from: "employees",
          localField: "employees",
          foreignField: "_id",
          as: "employeeDetails"
      }}
    ])

  Option B (Application-side join):
    1. Find shift, get employees array of ObjectIds
    2. Find employees where _id in that array

Performance: Embedded array + $lookup is generally faster than junction table
for read-heavy operations, but requires careful indexing.

8. RISKS AND COMPLICATIONS

RISK 1: Data Loss During Migration
-----------------------------------
Description: transform_db.js script fails mid-migration, leaving database
             in inconsistent state.
Mitigation: 
- Create full database backup before running transform_db.js
- Run migration in a test environment first
- Make script idempotent where possible (safe to run multiple times)
- Verify each step with Compass before proceeding

RISK 2: ObjectId String Conversion Errors
------------------------------------------
Description: Application receives string ID from URL/form, fails to convert
             to ObjectId properly, causing "document not found" errors.
Mitigation:
- Create centralized validation function: validateObjectId(string)
- Always wrap conversions in try-catch (invalid ObjectId throws error)
- Consistent use of new mongodb.ObjectId(id) throughout codebase

RISK 3: Session Management Complexity
-------------------------------------
Description: Without express-sessions, manual implementation may have
             security vulnerabilities (session fixation, hijacking).
Mitigation:
- Use cryptographically secure random session IDs (crypto.randomBytes)
- Implement proper HttpOnly, Secure, SameSite cookie attributes
- Store session expiration server-side, not just in cookie
- Regenerate session ID on login (prevent fixation)

RISK 4: Embedded Array Size Limits
----------------------------------
Description: MongoDB documents have 16MB limit. If a shift could have
             thousands of employees, employees array could exceed limit.
Likelihood: Low for scheduling system (shifts typically have < 50 employees)
Mitigation:
- Monitor array size in application logic
- Document assumption that shifts have reasonable employee counts
- If needed in future, could revert to referencing pattern

RISK 5: Photo Storage Performance
---------------------------------
Description: Storing binary image data in MongoDB documents may impact
             query performance and memory usage.
Options:
- Store as BSON Binary (GridFS if large)
- Store file path/URL, keep images on filesystem/S3
- Decision: [YOUR CHOICE HERE - justify based on expected photo size]

RISK 6: Security Log Collection Growth
--------------------------------------
Description: High-traffic system could generate massive security_log collection.
Mitigation:
- Implement log rotation/archival strategy (out of scope but note for future)
- Create index on timestamp for efficient querying
- Consider capped collection if storage is concern

RISK 7: Concurrent Update Conflicts
----------------------------------
Description: Two users simultaneously modifying same shift's employees array
             could cause race conditions.
Mitigation:
- Use MongoDB's atomic $push/$pull operations (document-level locking)
- For complex updates, consider findAndModify with version checking
- Acceptable risk for this assignment scope
